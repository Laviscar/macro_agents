# Macro Agents v1.9 — 时效性 (Timeliness / Freshness) 设计

> 状态:已与用户敲定(2026-06-11)。分支 `v1.9-dev`。解决"信息有多新、陈旧如何处理"的系统问题(挂服务器前的前置)。

## 背景 / 动机
用户挂常驻前的核心顾虑:不知道新闻和圆桌的**时效性**。诊断发现真问题是 **处理滞后**——抓取 >> 分析,且分析**最旧优先(FIFO)**,导致最新新闻堵在队列、世界树反映的是几天前的旧闻。业界对策:TTL / watermark / 时间衰减 / 取代(supersession)/ freshness 读数。本版补 4 件:

- **A. 待召开过期/取代**:同一资产出现新待召开请求时,该资产**旧的**待召开标记 `expired`(保留、UI 灰掉)。
- **B. 分析改最新优先**:常驻 loop 的 triage/analysis 改 newest-first,让最新新闻先进图、旧积压自然老化。
- **C. Freshness 读数**:一个函数报"最新新闻/证据/驱动切换/FRED 多旧、积压多少",让用户一眼看清时效(并在 run_loop 日志输出)。
- **D. memo 陈旧标记**:已召开圆桌 session 被同资产更新的 session 或更新的驱动切换取代、或超过 TTL → 标 `stale`(保留、UI 灰掉)。

**原则**:A/D 只标记不删除(保留记录);UI 的"灰掉"是已有委员会页的小改;**不做新 UI 页**(先功能后 UI,见 [[defer-ui-until-content-review]])。

---

## A. 待召开过期/取代(同一资产)
- `PendingConvocation` 加 `status: Literal["active","expired"] = "active"`。
- `CommitteeRepository.supersede_pending(asset_id, keep_created_at)`:把该资产 `status=active` 且 `created_at < keep_created_at` 的待召开改 `expired` 存回(保留文件)。
- `stages._evaluate_committee`:`evaluate_assets` 产出新 pending 后,对**本轮有新 pending 的每个资产**调 `supersede_pending(asset_id, 该资产新 pending 的 created_at)`,再 save 新的。
- `CommitteeRepository.list_pending()` 仍返回全部(active+expired);新增 `list_active_pending()`。
- 委员会页:expired 的灰显(无召开按钮、标"⏸ 已过期/被取代"),active 正常。

## B. 分析最新优先
- `build_run_loop` 非 run_now 分支:`triage_kwargs/analysis_kwargs` 加 `newest_first = os.environ.get("RUN_LOOP_NEWEST_FIRST","true").lower() != "false"`(默认 True)。
- 含义:最新新闻先筛选/分析 → 其证据(created_at=分析时刻)被整合 watermark 立即纳入 → 进图。旧的未分析新闻自然老化(不强求消化全部积压——freshness 优先于 completeness)。

## C. Freshness 读数
- `pipelines/freshness.py::freshness_summary(news_repo, graph_repo, committee_repo, fred_repo=None, now=None) -> dict`:
  - `latest_news_at` / `latest_evidence_at` / `latest_shift_at` / `oldest_active_pending_at` / `latest_fred_date`
  - `pending_sort_backlog` / `pending_analysis_backlog` / `active_pending_count`
  - 各时间 + `*_age_minutes`(相对 now)。
- `run_loop`:加一个轻量 `freshness` stage(或 tick 时)`log_event` 输出该摘要(0 LLM)。UI 展示留后。

## D. session 陈旧标记
- `committee/staleness.py::session_is_stale(session, all_sessions, driver_shifts, ttl_hours=None, now=None) -> bool`:
  - 同资产存在 `created_at` 更新的 session → True;或
  - 该资产存在 `at > session.created_at` 的 driver_shift(局势已变)→ True;或
  - `ttl_hours` 给定且 session 超龄 → True。
- `presenters/committee_presenter.build_committee_view`:给每个 session 附 `stale: bool`(用 `session_is_stale`)。
- 委员会页历史区:stale 的灰显 + 标"⚠️ 可能已过时(局势已变)"。

## 数据 / 兼容
- 仅加字段(PendingConvocation.status,默认 active;旧文件读出即 active)+ 新模块/方法。不改图引擎、不改 strength/边。
- env:`RUN_LOOP_NEWEST_FIRST=true`、(可选)`COMMITTEE_SESSION_TTL_HOURS`(D 的 TTL,默认不启用纯靠取代判断)。

## 测试
- A:supersede_pending 把旧 active 改 expired、新 active 保留;_evaluate_committee 集成后旧 pending 过期。
- B:build_run_loop 的 triage/analysis kwargs 含 newest_first=True(默认)。
- C:freshness_summary 字段齐全、age 计算正确(注入 now)。
- D:session_is_stale 三条件(更新 session / 更新 shift / TTL)+ presenter 标 stale。
